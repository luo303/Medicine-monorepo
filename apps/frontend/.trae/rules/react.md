# React 性能优化规则

## 一、核心原则

1. 优化核心是**避免不必要的组件重渲染与重复计算**
2. 先定位性能瓶颈再针对性优化，拒绝无脑过度优化，所有优化手段均有执行成本
3. 代码可读性优先，不做无意义的性能封装
4. 最小化状态更新影响范围，状态下沉至真正使用的组件

## 二、核心性能规则

1. 传递给子组件的函数props，需用`useCallback`保证引用稳定，避免子组件无效重渲染；仅子组件用`React.memo`、函数作为Hook依赖等场景必须使用。
2. 数组遍历转换、复杂对象处理等计算密集型操作，需用`useMemo`缓存结果；简单单行计算、极小数据量场景无需使用。
3. 仅依赖props、无副作用的纯展示组件（尤其列表项、复杂DOM结构组件），需用`React.memo`包裹，需配合引用稳定的props使用，否则优化失效。
4. `useEffect`依赖数组必须包含所有内部使用的外部变量，禁止用其实现渲染期可直接计算的派生状态，杜绝无效state+effect链式更新。
5. 传递给子组件的对象/数组props，需用`useMemo`缓存或提取至组件外，保证引用稳定。
6. Context必须按更新频率、功能模块拆分，禁止全量状态打包，避免单次更新引发全量组件重渲染。
7. 列表渲染必须设置稳定唯一的key，优先使用业务/后端唯一ID，禁止用index、随机值作为key。
8. 超100项的长列表必须使用虚拟滚动，仅渲染视口内列表项，降低DOM渲染开销。
9. 非首屏图片必须实现懒加载，优先使用浏览器原生`loading="lazy"`，优化首屏性能。
10. 搜索输入、滚动、拖拽等高频触发事件，需按场景做防抖/节流处理，减少渲染次数。
11. 复杂表单字段状态需精细化管理，优先独立拆分或用`useReducer`，禁止全量字段打包到单个state。

## 三、提交前检查清单

- 函数/对象/数组props已保证引用稳定，合理使用`useCallback`、`useMemo`
- 纯展示组件合理使用`React.memo`，`useEffect`依赖完整无滥用
- Context已规范拆分，列表使用稳定唯一key
- 长列表已做虚拟滚动，非首屏图片已懒加载
- 高频事件已做防抖/节流，表单状态已精细化管理
- 已通过React DevTools Profiler验证无无效重渲染

## 四、避坑误区

1. 禁止过度优化，缓存API仅针对明确性能瓶颈使用
2. 规避`useCallback`闭包陷阱，依赖数组必须完整
3. React 18已自动批处理同步状态更新，异步回调中可合并更新减少渲染
4. 禁止过度状态提升，严格遵循状态最小化原则
